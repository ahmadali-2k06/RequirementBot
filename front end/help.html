3. The Precisely Workflow

Follow this structured process to generate a complete Software Requirement Specification (SRS).

Step Action Detailed Description
1 Create Project Click the New Project button. Enter a name (e.g., "Hospital Management System") and a short description. This creates a new container for your data.
2 Define Scope (Wizard) The bot will automatically trigger the Scope Wizard. It will ask you 6-7 questions to understand the context.
You will define: Primary Users, Key Features, Exclusions, and Technical Constraints.
Note: Constraints entered here (e.g., "Must use Python") are automatically split and added to the Right Sidebar under the "Constraints" tab.
3 Gather Requirements Once the scope is set, type your requirements into the chat input.
Example: "The system shall generate a monthly PDF report."

The AI will immediately:
  • Classify it as Functional or Non-Functional.
  • Check for ambiguity (e.g., "Is 'fast' specific enough?").
  • Verify it aligns with the defined Scope.
If valid, it appears in the Chat AND the Right Sidebar.
4 Generate Report When you are finished, type the command exit in the chat.
The system will compile a Final HTML Report inside the chat window. This report includes:
  • Project Scope & Constraints
  • Summary Statistics (High/Medium/Low priority counts)
  • The full, formatted list of requirements
The chat input will be disabled to indicate the session is closed.

4. Chat Commands

You can control the bot using these specific keywords during the gathering phase.

Command Description
scope Prints the current project boundaries (Users, Features, Exclusions) to the chat. Helpful if you forget what is out of scope.
checklist Displays the "Requirement Quality Checklist" to remind you of best practices (SMART criteria).
summary Shows a real-time statistical breakdown (e.g., "Total: 15 | High Priority: 5 | Functional: 10").
exit Generates the Final Report and ends the current session.

5. The SMART Criteria

The AI Ambiguity Detector is trained to ensure your requirements meet these five standards.

  • S - Specific: Clear and unambiguous. (Bad: "Make it fast" vs Good: "Load in 2s")
  • M - Measurable: Can be tested with a Pass/Fail criterion.
  • A - Achievable: Realistic within technical constraints.
  • R - Relevant: Aligns with the business goals defined in the Scope.
  • T - Testable: A QA engineer can write a test case for it.
